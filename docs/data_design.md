# WasteWise - Data Design Document

## Database: Firebase Firestore (NoSQL Document Database)

---

## 1. Collections Overview

The database is organized into the following **Firestore Collections**:

| Collection Name | Purpose | Document ID |
|----------------|---------|-------------|
| `citizens` | Citizen user profiles | User UID (from Firebase Auth) |
| `drivers` | Driver user profiles | User UID (from Firebase Auth) |
| `admins` | Admin user profiles | User UID (from Firebase Auth) |
| `users` | Legacy/fallback user collection | User UID |
| `pickupRequests` | Waste pickup requests from citizens | Auto-generated |
| `schedules` | Regular collection schedules (common + personal) | Auto-generated |
| `routes` | Driver route assignments | Auto-generated |
| `tasks` | Driver task assignments linked to pickup requests | Auto-generated |
| `vehicles` | Vehicle fleet information | Auto-generated |
| `reports` | Issue/complaint reports from citizens | Auto-generated |
| `notifications` | User notifications | Auto-generated |
| `wasteTypes` | Waste type catalog (configurable) | Auto-generated |
| `statistics` | Daily aggregated statistics | Date string (YYYY-MM-DD) |

## 3. Relationships

### 3.1 User Relationships

- **Citizen** → **PickupRequest**: One-to-Many (`citizenId`)
- **Citizen** → **Report**: One-to-Many (`reporterId`)
- **Citizen** → **Notification**: One-to-Many (`userId`)
- **Citizen** → **Schedule**: One-to-Many (`citizenId`, if `isCommon = false`)

- **Driver** → **PickupRequest**: One-to-Many (`driverId`)
- **Driver** → **Route**: One-to-Many (`driverId`)
- **Driver** → **Task**: One-to-Many (`driverId`)
- **Driver** → **Vehicle**: Many-to-One (`assignedDriverId` in vehicles)
- **Driver** → **Schedule**: One-to-Many (`driverId`)

- **Admin** → **Report**: One-to-Many (`assignedTo`)

### 3.2 Pickup Request Relationships

- **PickupRequest** → **Citizen**: Many-to-One (`citizenId`)
- **PickupRequest** → **Driver**: Many-to-One (`driverId`)
- **PickupRequest** → **Route**: Many-to-One (`pickupRequestIds` array in Route)
- **PickupRequest** → **Task**: One-to-One (`requestId` in Task)
- **PickupRequest** → **Schedule**: One-to-One (`pickupRequestId` in Schedule)

### 3.3 Route Relationships

- **Route** → **Driver**: Many-to-One (`driverId`)
- **Route** → **PickupRequest**: One-to-Many (`pickupRequestIds` array)
- **Route** → **Schedule**: One-to-Many (`routeId` in Schedule)

### 3.4 Task Relationships

- **Task** → **PickupRequest**: Many-to-One (`requestId`)
- **Task** → **Driver**: Many-to-One (`driverId`)

## 4. Data Access Patterns

### 4.1 Common Queries

1. **Get citizen's pickup requests**:
   ```
   pickupRequests.where('citizenId', '==', userId).orderBy('requestedDate', 'desc')
   ```

2. **Get driver's assigned pickups**:
   ```
   pickupRequests.where('driverId', '==', driverId).where('status', '==', 'assigned')
   ```

3. **Get citizen's schedules** (common + personal):
   ```
   schedules.where('isCommon', '==', true)
   schedules.where('citizenId', '==', userId)
   ```

4. **Get unread notifications**:
   ```
   notifications.where('userId', '==', userId).where('isRead', '==', false)
   ```

5. **Get active drivers**:
   ```
   drivers.where('isOnline', '==', true).where('isAvailable', '==', true)
   ```

6. **Get open reports**:
   ```
   reports.where('status', '==', 'open').orderBy('createdAt', 'desc')
   ```

## 5. Security Rules (Firestore)

**Note**: These are conceptual rules. Actual implementation should be configured in Firebase Console.

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Citizens can read/write their own data
    match /citizens/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
      allow read: if request.auth != null && get(/databases/$(database)/documents/admins/$(request.auth.uid)).data.role == 'admin';
    }
    
    // Drivers can read/write their own data
    match /drivers/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
      allow read: if request.auth != null && get(/databases/$(database)/documents/admins/$(request.auth.uid)).data.role == 'admin';
    }
    
    // Admins can read/write all admin documents
    match /admins/{userId} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
    
    // Pickup requests: Citizens can create/read their own, Drivers can read assigned, Admins can read all
    match /pickupRequests/{requestId} {
      allow create: if request.auth != null;
      allow read: if request.auth != null && (
        resource.data.citizenId == request.auth.uid ||
        resource.data.driverId == request.auth.uid ||
        get(/databases/$(database)/documents/admins/$(request.auth.uid)).data.role == 'admin'
      );
      allow update: if request.auth != null && (
        resource.data.driverId == request.auth.uid ||
        get(/databases/$(database)/documents/admins/$(request.auth.uid)).data.role == 'admin'
      );
    }
    
    // Similar rules for other collections...
  }
}
```

## 6. Data Migration Notes

- The `users` collection is kept for **legacy compatibility**.
- New users are stored in role-specific collections (`citizens`, `drivers`, `admins`).
- The `DatabaseService` checks all collections when looking up a user.

---

## 7. Performance Considerations

1. **Composite Indexes**: Create indexes for all query combinations used in the app (especially for `pickupRequests`, `schedules`, `notifications`).
2. **Pagination**: Use `limit()` and `startAfter()` for large result sets.
3. **Real-time Listeners**: Use Firestore streams for live updates (already implemented in `DatabaseService`).
4. **Caching**: Firestore automatically caches data offline (enable offline persistence in Flutter).

---

## 8. Backup & Recovery

- Firestore provides automatic backups (configure in Firebase Console).
- Export data periodically using Firestore export functionality.
- Consider implementing data retention policies for old records.
